# 그릴에이전트

## 목적

`grill-with-docs`의 원래 질문 규칙은 그대로 사용한다. 이 스킬은 그 규칙을 바꾸지 않고, 질문자 서브에이전트와 답변자 서브에이전트를 분리해 메인 에이전트가 중간에서 릴레이하는 운영 방식만 정의한다.

## 전제

- 먼저 `grill-with-docs` 스킬 본문을 읽고, 그 문안을 우선 규칙으로 삼는다.
- 이 스킬은 사용자가 서브에이전트 방식, 에이전트 오케스트레이션, 또는 그릴에이전트를 명시했을 때 사용한다.
- 대상 문서가 현재 source of truth인지 확인한 뒤 시작한다.
- 계획 주제는 프로젝트의 planning authority index나 GBrain canonical page를 먼저 읽고, 그 뒤에 `CONTEXT.md`, ADR, 관련 코드 경로를 확인한다.
- 로컬 문서 미러, 삭제된 repository 계획서, 오래된 masterplan, 오래된 microplan은 현재 canonical source가 명시적으로 채택한 경우에만 현재 근거로 사용한다.
- 서브에이전트가 직접 파일을 수정하지 않는다.
- 합의된 변경은 메인 에이전트만 현재 source of truth에 반영한다. 대상이 파일이면 파일 편집 도구(Claude Code `Edit` / Codex `apply_patch`)를 쓰고, 대상이 GBrain 같은 외부 canonical source이면 그 source를 갱신하거나 필요한 갱신 내용을 정확히 보고한다.

## 토큰과 성능

- 서브에이전트는 한 번 띄운 뒤 재사용한다. 매 라운드마다 새로 띄우지 않는다.
- 메인 에이전트는 시작 전에 `context packet`을 만든다. 여기에는 현재 source of truth, 관련 맥락 source, 현재 대상 라운드, 이미 합의된 결정만 넣는다.
- 긴 문서 전문을 서브에이전트에게 붙여 넣지 않는다. 파일 경로를 주고 필요한 부분은 서브에이전트가 직접 읽게 한다.
- 매 라운드 입력에는 직전 질문, 직전 답변, 누적 결정 요약만 포함한다. 전체 대화 로그를 반복 전달하지 않는다.
- 누적 결정 요약은 10개 항목을 넘기지 않는다. 넘으면 메인 에이전트가 “현재 구현 판단에 필요한 결정”만 남겨 압축한다.
- 질문자와 답변자의 출력은 각각 4개 필드 형식만 허용한다. 근거는 핵심 파일/라인 중심으로 제한한다.
- 같은 파일을 두 에이전트가 반복해서 크게 읽고 있으면 메인 에이전트가 필요한 범위를 좁혀 다음 라운드에 전달한다.
- 토큰 절감과 성능 최적화는 검증 범위를 줄이는 방식으로 하지 않는다. 최적화는 반복 전달을 줄이고 필요한 파일 범위를 좁히는 데만 쓴다.

## 닫힘 우선 원칙

- 이 스킬의 목표는 대상 문서를 최대한 명확하게 닫고, 구현 전에 빠진 질문을 남기지 않는 것이다.
- 대상 문서 하나를 질문 하나로 끝내지 않는다.
- 같은 대상 문서에 대해 질문자가 더 이상 문서·코드 근거의 의미 있는 질문이 없다고 선언할 때까지 라운드를 계속한다.
- 다음 대상 문서로 넘어가는 것은 현재 대상의 열린 질문이 소진된 뒤에만 가능하다.
- 성능을 이유로 라운드 수를 제한하지 않는다.
- 같은 쟁점이 반복되면 종료하지 말고, 반복 원인을 요약해 `needs-user` 또는 `blocker`로 분류한다.
- 질문이 소진되었다는 선언은 `Meaning`, `Scope`, `Order`, `Close` 렌즈를 모두 확인한 뒤에만 허용한다.

## 역할

### 메인 에이전트

- 대상 문서와 현재 라운드 상태를 관리한다.
- `context packet`과 누적 결정 요약을 관리한다.
- 질문자에게 다음 질문 하나를 요청한다.
- 답변자에게 그 질문을 전달한다.
- 답변을 `resolved`, `doc-change`, `needs-user`, `blocker` 중 하나로 분류한다.
- `doc-change`는 사용자 승인이 필요 없는 합의된 문서 정정일 때만 직접 반영한다.
- `needs-user`나 `blocker`는 멈추고 사용자에게 짧게 묻는다.
- 현재 대상 문서의 질문이 소진되기 전에는 다음 대상 문서로 넘어가지 않는다.

### 질문자 서브에이전트

- `grill-with-docs` 방식으로 정확히 질문 하나만 만든다.
- 질문은 `CONTEXT.md`, ADR, 계획서, 기존 코드, 또는 문서화된 도메인 용어를 쓰는 구체 시나리오에 근거해야 한다.
- 코드로 확인 가능한 것은 먼저 확인하고, 확인 결과를 근거에 포함한다.
- 추천 답변을 함께 제시한다.
- 파일을 수정하지 않는다.
- 더 이상 질문이 없다고 판단할 때는 새 질문 대신 소진 선언을 반환한다. 소진 선언에는 확인한 렌즈와 남은 열린 쟁점이 없다는 근거를 포함한다.

반환 형식:

```text
질문: ...
근거: ...
추천 답변: ...
답변자가 확인해야 할 지점: ...
```

소진 선언 형식:

```text
질문: 없음
근거: Meaning/Scope/Order/Close에서 남은 문서·코드 근거 질문이 없음
추천 답변: 현재 대상은 다음 대상으로 넘어가도 됨
답변자가 확인해야 할 지점: 소진 판단에 누락된 렌즈가 있는지
```

### 답변자 서브에이전트

- 질문자가 낸 질문 하나에만 답한다.
- 문서와 코드 근거로 결론을 낸다.
- 질문자의 추천 답변에 `동의`, `수정 동의`, `반대`, `판단 보류` 중 하나로 응답한다.
- 문서와 코드가 충돌하면 충돌을 명확히 표시한다.
- 파일을 수정하지 않는다.

반환 형식:

```text
결론: ...
근거: ...
구현 전 조치: ...
판정: 동의|수정 동의|반대|판단 보류
```

## 라운드 절차

1. 메인 에이전트가 대상 문서, 관련 맥락 문서, 이미 합의된 결정을 정리한다.
2. 메인 에이전트가 짧은 `context packet`을 만든다.
3. 질문자와 답변자 서브에이전트를 각각 한 번 띄운다.
4. 두 서브에이전트에는 파일 전문 대신 `context packet`과 필요한 파일 경로만 전달한다.
5. 질문자에게 현재 대상에 대한 다음 질문 하나를 요청한다.
6. 답변자에게 질문 전체와 질문자의 추천 답변을 전달한다.
7. 메인 에이전트가 답변을 분류하고 사용자에게 짧게 보고한다.
8. 문서 변경이 합의되면 메인 에이전트가 반영하고, 필요한 focused 검증을 실행한다.
9. 같은 대상 문서에 대해 질문자에게 다음 질문을 다시 요청한다.
10. 질문자가 소진 선언을 반환하고, 답변자 또는 메인 에이전트가 누락 렌즈를 발견하지 못하면 다음 대상 문서로 넘어간다.

`context packet` 형식:

```text
Target: ...
Source of truth: ...
Context sources: ...
Relevant code: ...
Resolved decisions: ...
Current question focus: ...
Stop condition: ...
```

라운드 입력 형식:

```text
Round: ...
Target: ...
Resolved decisions since last round: ...
Previous question/answer summary: ...
Ask/answer exactly one question.
```

## 대상별 렌즈

대상 문서의 종류에 따라 보조 렌즈를 하나만 추가할 수 있다. 보조 렌즈는 질문자가 다음 질문을 고를 때만 사용하고, `grill-with-docs`의 한 질문씩, 문서 근거, 코드 확인 규칙보다 앞서지 않는다.

마이크로플랜을 구울 때는 `microplan-review`의 네 기준을 질문 선택 렌즈로 쓴다:

- `Meaning`: 의도 문서나 도메인 계약과 충돌하는가?
- `Scope`: 하나의 의미 축이며 구현 잎 작업이 3개 이하인가?
- `Order`: 완료된 이전 단계에만 의존하고, 후속 단계 일을 끌어오지 않는가?
- `Close`: 완료 게이트가 1-3개의 focused check로 관찰 가능한가?

마이크로플랜 렌즈 사용 시에도 `PASS`, `FIX`, `ESCALATE` 판정 보고서를 만들지 않는다. 필요한 경우 마지막 요약에서만 짧게 판정한다.

## 중단 조건

- 사용자 판단이 필요한 도메인 선택지가 나오면 멈춘다.
- 질문이 원래 `grill-with-docs` 범위를 벗어나 제품 아이디어, 취향 토론, 구현 방식 선호 논쟁으로 흐르면 멈춘다.
- 문서와 코드의 충돌이 구현 전 결정 없이는 해소되지 않으면 멈춘다.
- 외부 작업, 자격증명, 비용 발생, 운영 배포가 필요하면 멈춘다.

## 신선함 유지

관점의 신선함은 새 절차를 덧붙여 만들지 않는다. 매 라운드마다 원래 `grill-with-docs` 문안에 따라 다음으로 가장 중요한 문서·용어·코드·시나리오 경계를 묻는다.

역할 교대가 필요하다고 판단되면 한 번만 수행할 수 있다. 이때도 두 에이전트 모두 `grill-with-docs`의 원래 질문 규칙을 그대로 따른다.

## 금지

- `grill-with-docs`의 질문 철학을 재작성하지 않는다.
- 한 라운드에 여러 질문을 묶지 않는다.
- 서브에이전트가 문서나 코드를 수정하지 않는다.
- 문서 근거 없는 상상 리스크나 새 기능 제안을 만들지 않는다.
- 합의되지 않은 내용을 `CONTEXT.md`나 ADR에 쓰지 않는다.
