# 코드 리뷰

머지 전에 고쳐야 하는 실제 계약 결함만 찾는다.
재현 가능한 결함을 놓치지 않는 것보다 근거 없는 의혹을 finding으로 승인하지 않는 일을 우선한다.

## 범위

리뷰를 한 번만 수행한다.

- 코드를 수정하지 않는다.
- 재리뷰 루프를 실행하지 않는다.
- `code-review-loop`를 호출하지 않는다.

## 핵심 기준

- 검사에서 나온 의혹은 finding이 아닌 `후보 의혹`으로 다룬다.
- 상위 구조는 대상 계약을 해석할 때만 사용한다.
- 상위 구조를 리뷰·수정 범위를 넓히는 근거로 사용하지 않는다.
- finding은 변경 줄에서 잘못된 출력·효과까지 이어지는 실행 가능한 인과 사슬로 증명한다.
- 신뢰도·심각도·관점 수를 finding의 승인 근거로 사용하지 않는다.
- 후보를 승인하기 전에 인용된 코드와 계약을 직접 확인한다.

## 리뷰 계약 잠금

검사를 시작하기 전에 아래 `리뷰 계약`을 근거와 함께 작성하고 고정한다.

1. **원천**: 사용자 목적, 승인된 계획·PR 설명·합의, 적용 지침, 변경 전 코드 계약
2. **상위 목적·역할**: 대상이 상위 구조에서 맡는 이유와 책임
3. **리뷰 경계**: base/head 또는 로컬 diff, 변경 파일·줄·심볼, 직접 호출자·피호출자, 공유 데이터 계약
4. **입력 계약**: 데이터 포맷·범위, 사전조건, 권한, 시작 상태
5. **절차 계약**: 처리 순서, 상태 전이, 분기, 부작용, 오류·자원 생명주기
6. **출력 계약**: 반환값·예외, 저장 상태, 외부 효과, 직접 소비자가 관측하는 결과
7. **보호 불변조건**: 변경 뒤에도 유지해야 하는 기존 성공 동작
8. **비목표·금지 범위**: 이번 변경이 해결하지 않는 영역과 수정 권한이 없는 구조
9. **검증 근거**: 계약 위반을 확인하거나 기각할 코드·테스트·명세 위치

상위 구조는 2번의 의미를 확인할 때만 사용한다.
상위 파일이나 서비스가 3번 경계에 없으면 해당 구조의 개선점을 finding으로 승인하지 않는다.

필수 계약을 코드·계획·지침에서 확인할 수 없으면 추측하지 않는다.
그 답이 후보 승인 여부를 바꾼다면 `material contract question`으로 분류한다.
그 불확실성에 의존하는 후보는 finding으로 승인하지 않는다.

## 준비

1. `git status --short`와 리뷰 대상을 확인한다.
2. PR이면 번호, base/head, 변경 파일, diff, 기존 댓글, 상태를 확인한다.
3. 선택한 범위에 리뷰할 diff가 없을 때만 보고하고 중단한다.
4. PR 상태·생성 방식·변경 크기·이전 리뷰 여부로 명시적인 리뷰 요청을 취소하지 않는다.
5. 변경 파일에 적용되는 지침을 확인한다.
6. 리뷰 계약을 잠근다.
7. 실행할 관점과 각 관점이 확인할 계약 질문을 기록한다.
8. 계약 질문을 만들 수 없는 선택 관점은 실행하지 않는다.

## 리뷰 관점

관점은 잠긴 계약에서 증거를 찾는 도구로만 사용한다.

### 기본 관점

1. **변경 diff**
   - 변경 줄과 가까운 문맥에서 입력·절차·출력 계약의 직접 이탈을 찾는다.
2. **1차 영향범위**
   - 직접 호출자·피호출자와 같은 데이터 계약의 직접 소비자를 확인한다.
   - 변경으로 깨지는 연결만 찾는다.
   - 1차 영향범위 밖으로 넓히지 않는다.

### 선택 관점

계약 질문을 먼저 정한 경우에만 아래 관점을 실행한다.

- **지침 준수**: 현재 계약 판정을 바꿀 수 있는 적용 지침 조항을 확인한다.
- **Git 이력**: 현재 원천으로 해소되지 않은 모호성만 과거 변경 이유로 보충한다.
- **코드 주석**: 변경 코드에 적용되는 동작 제약을 확인한다.

표현의 논리성을 후보의 증거로 사용하지 않는다.

## 필요 시 집중 검증

코드와 계약만으로 후보의 도달 가능성이나 잘못된 출력·효과를 확정할 수 없을 때만 검증한다.
적용 지침이 허용하는 로컬 검사로 해당 질문만 판정한다.
가장 좁은 범위로 아래 내용을 기록하고 실행한다.

1. 판정할 리뷰 계약 항목과 질문
2. 실행할 명령과 범위
3. 기대 결과
4. 실제 출력

잠긴 리뷰 경계 안에서 외부 상태를 바꾸지 않는 방식으로 검증한다.
허용된 검증으로도 확정할 수 없으면 finding을 승인하지 않는다.

## 후보 의혹 형식

후보마다 아래 항목을 모두 작성한다.

1. **위반 계약**: 잠긴 리뷰 계약의 정확한 항목
2. **변경 원인**: 원인이 되는 변경 파일·줄·심볼
3. **도달 가능한 입력·상태**: 현재 코드에서 실제 도달 경로가 있는 사례
4. **실패 인과 사슬**: 변경 줄 → 절차·상태 이탈 → 실패 전파
5. **잘못된 출력·효과**: 계약과 달라지는 반환·예외·저장·외부 효과·사용자 관측 결과
6. **직접 증거**: 코드 경로, 계약 원천, 테스트 또는 직접 소비자 근거
7. **diff 인과성**: 변경 전에는 성립하지 않았고 이번 diff 때문에 생긴 이유
8. **경계 증명**: 리뷰 경계와 1차 영향범위 안에 있다는 이유

필수 항목이 하나라도 비거나 추정뿐이면 후보 의혹으로도 보고하지 않는다.
finding을 승인하기 전에는 수정 방향을 작성하지 않는다.

지배 계약을 확인할 수 없어 1번을 채울 수 없고 그 답이 승인 여부를 바꾼다면 후보를 폐기하지 않는다.
해당 후보를 `material contract question`으로 분류한다.

## Finding 승인

각 후보의 인용 위치를 직접 읽고 분류한다.

| 분류 | 판정 | 처리 |
|---|---|---|
| 계약 안의 결함 | 모든 필수 증거가 있고 현재 diff가 계약을 실제로 깸 | finding 승인 |
| 비실행·추측 | 도달 경로나 잘못된 결과가 증명되지 않음 | 폐기 |
| 기존 문제·범위 밖 | 변경 전부터 존재하거나 리뷰 경계 밖임 | 폐기 |
| material contract question | 지배 계약이 없고 그 답이 승인 여부를 바꿈 | 정확한 질문을 열린 질문으로 보고 |
| 계약 변경 필요 | 수정하려면 입력·절차·출력 또는 제품 결정을 바꿔야 함 | 열린 질문으로 분리 |
| 전제 충돌 | 후보 전제가 원천·기존 성공 계약과 모순됨 | 폐기하고 최종 출력에서 제외 |

숫자 신뢰도는 사용하지 않는다.
심각도는 승인한 finding의 정렬에만 사용한다.

승인한 뒤에만 잠긴 리뷰 계약과 직접 영향범위 안에서 최소 수정 방향을 작성한다.
수정에 새 실행 경로·분기·제품 계약 승인이 필요하면 `계약 변경 필요`로 분류한다.

## 자동 탈락

- 가능성, 미래 확장성, 일반적 방어, 선호 스타일만 근거로 한 주장
- 상위 아키텍처를 읽고 현재 경계 밖 구조를 바꾸자는 주장
- 변경하지 않은 줄에만 존재하는 기존 문제
- 직접 소비자까지 이어지지 않는 가상 영향
- 명시적 계약과 연결되지 않은 일반 보안·문서·테스트 커버리지 주장
- 인과 증명 없이 CI·import·타입·포맷·깨진 테스트 신호만 다시 말한 주장
- 후보가 주장하는 잘못된 출력·효과가 승인된 원천에 명시된 의도적 동작 변경인 경우
- 특정 해결책을 먼저 정하고 만든 문제 정의

## 출력

1. 승인한 finding을 심각도 순서로 보고한다.
   - 파일·줄
   - 위반 계약
   - 입력부터 잘못된 결과까지의 인과 사슬
   - 직접 증거
   - 잠긴 리뷰 계약과 직접 영향범위 안의 최소 수정 방향
2. `material contract question`과 계약 변경이 필요한 항목을 열린 질문으로 분리한다.
3. 리뷰 경계와 확인하지 못한 계약을 짧게 남긴다.

승인한 finding이 없으면 finding 구역만 아래 문구로 바꾼다.

```md
### 코드 리뷰

승인된 finding이 없습니다.
```

열린 질문과 확인하지 못한 계약이 있으면 해당 구역은 그대로 출력한다.
