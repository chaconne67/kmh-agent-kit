- GBrain을 먼저 읽는다 (실행법·페이지 목록은 `## GBrain` 참조).

- 필요한 정보가 없으면 오류로 중단한다. 필요한 정보를 임의로 만들어 넣거나 성공한 것처럼 처리하지 않는다. 없어도 되는 정보는 시스템에 미리 정해 둔 값만 사용한다.
- 스킬 수정, 제작, 삭제 관련 작업에 `$superpowers:writing-skills` 스킬 **사용 금지**

## GBrain
- CLI는 `~/.gbrain/bin/gbrain_with_google_env.sh`로 실행한다.
- 읽기 `get <slug>` · 키워드 `search "<query>"` · 의미검색 `query`/`ask "<question>"` · 목록 `list`. (`show` 없음)
- 작업 전 먼저 읽는다: `project/exdigm-operating-context`, `agent/gbrain-operating-protocol`, `project/gbrain-agent-memory-loop`
- GBrain이 안 되면 작업하지 말고 실패를 보고한다.
- 코드와 GBrain이 다르면 **코드가 기준**. 검증 후 GBrain을 갱신한다.

## 일괄 승인 게이트

- 변경 작업의 승인 요청은 한 번만 한다.
- 통합 계획 문서를 작성해 일괄 승인을 요청하고, 승인 후에는 구현·검증·리뷰·문서화·커밋까지 추가 승인 없이 진행한다.
- 승인된 목표·범위 또는 부작용이 달라질 때만 새 승인을 요청한다.

## 최종 경로 하드 게이트 (SSP)

### 용어

- **최종 경로(SSP, Single Success Path)**: 작업물이 실제로 성공을 만드는 단 하나의 공식 실행 경로다. 이 문서의 「최종 경로」는 모두 이 뜻이다.
- **고아 패치**: 최종 경로의 어느 단계에도 병합되지 않은 변경이다.
- **순증가**: 대체·병합·삭제할 기존 요소 없이 파일·분기·LLM 호출·프롬프트 블록·코드량만 늘리는 변경이다.
- **엔트로피**: 최종 경로를 이해·실행·검증할 때 함께 추적해야 하는 요소와 연결 관계의 복잡도다. 경로·분기·파일·함수·LLM 호출·프롬프트 규칙·임시 패치·보정 단계가 늘수록 높아진다.

### 적용 조건

- 입력에서 최종 결과까지 이어지는 반복 실행 경로 자체를 새로 만들거나 분석·테스트·수정할 때만 적용한다. 개별 파일이나 문구를 고친다는 이유만으로 적용하지 않는다.

### 의도

- 언어모델은 탐색 중에 새 분기·보정·임시 패치를 계속 추가하는 경향이 있다. 이를 방치하면 산출물의 엔트로피가 계속 증가한다.
- SSP의 목적은 한 사례를 임시로 성공시키는 것이 아니다. 요구사항과 정확성을 지키면서 하나의 최종 경로로 통합하고, 산출물의 엔트로피를 필요한 최소 수준까지 낮추는 것이다.

### 원리

1. **하나의 경로**: 성공을 만드는 공식 경로는 하나만 유지한다.
2. **경로 안에 병합**: 모든 영구 변경은 최종 경로의 특정 실행 단계에 들어가야 한다.
3. **엔트로피 최소화**: 같은 요구사항·정확성·명확성을 충족한다면 경로·분기·파일·함수·LLM 호출·프롬프트 규칙·보정 단계가 가장 적은 해결책을 선택한다. 새 요소는 기존 요소로 해결할 수 없고 전체 엔트로피를 낮출 때만 만든다. 수정 후에도 실행 순서가 한눈에 읽혀야 한다.
4. **사례 일반화**: 특정 사례의 값·표현·산출물을 넣지 않는다. 반복 가능한 구조·계약·역할의 문제만 반영한다.
5. **같은 경로로 검증**: 수정한 최종 경로를 직접 실행한 결과만 성공으로 인정한다.

### 실행 절차

1. **단계 선언**: 분석·테스트·수정 중 현재 단계를 먼저 밝힌다. 분석·테스트 중에는 최종 경로를 바꾸지 않는다.
2. **수정 전 잠금**: 수정 전에 아래 네 가지를 구체적으로 밝힌다. 하나라도 밝히지 못하면 수정하지 않는다.
   - ① 현재 최종 경로: 주요 파일·함수·LLM 호출을 실제 실행 순서대로 적는다.
   - ② 병합 위치: 이번 변경이 들어갈 실행 단계를 적는다. 병합할 단계가 없으면 수정하지 않는다.
   - ③ 대체·정리 대상: 고치거나 병합하거나 삭제할 기존 요소와, 변경 후 엔트로피가 줄거나 유지되는 이유를 적는다. 기존 요소가 없으면 순증가로 판정한다.
   - ④ 검증 방법: 변경 후 최종 경로를 실행할 명령 또는 실제 진입점과 기대 결과를 적는다.
3. **수정**: ①~④에서 밝힌 범위 안에서만 최종 경로를 고친다. 조건을 충족하지 못하면 코드를 추가하지 말고 더 작은 변경으로 다시 설계한다.
4. **임시 코드 정리**: 조사·진단 결과를 최종 경로에 병합하고 임시 코드·파일을 삭제한다. 끝내지 못하면 미완료다.
5. **검증**: ④에서 밝힌 방법으로 최종 경로를 직접 실행한다.
6. **실패 복구**: 검증이 실패하면 이번 작업에서 병합한 미검증 수정만 즉시 되돌리고 분석 단계로 돌아간다.

- **신규 작성**: 최종 경로가 없다면 ①과 ④를 먼저 밝히고, 이를 최초의 최종 경로로 삼는다. 이후 수정부터 ①~④를 모두 적용한다.
- **단순 변경**: 원인과 결과를 화면·로그·테스트 결과로 바로 확인할 수 있을 때만, 수정 이유·병합 위치·검증 방법을 한 문장으로 밝히고 진행할 수 있다.

### 금지

- 고아 패치를 만들지 않는다.
- 사용자 승인 없이 순증가하지 않는다.
- 선언하지 않은 단계·분기·별도 실행 도구(runner)·보정 흐름을 추가하지 않는다.
- 특정 사례만 통과시키는 값·표현·산출물을 최종 경로에 넣지 않는다.
- 테스트·진단·임시 방어 코드를 최종 경로에 남기지 않는다.
- 임시 스크립트·수동 보정·사후 보정·우회 경로에서 얻은 성공을 검증 통과로 인정하지 않는다.

### 통과 조건

- 변경한 최종 경로를 직접 실행해 성공해야 한다.
- 같은 원인에서 나온 실패가 검증 대상에서 다시 발생하지 않아야 한다.
- 기존에 통과하던 사례도 같은 최종 경로로 계속 통과해야 한다.
- 경로·분기·중복 규칙·보정 단계가 불필요하게 늘지 않아야 한다. 불가피하게 늘어나는 경우에는 수정 전에 사용자 승인을 받아야 한다.
- 임시 코드와 파일이 남아 있지 않아야 한다.
- 한 사례의 성공이나 임시 성공만으로는 통과하지 못한다.

## 출력 형식 및 설명 방식 (반드시 지킴)

### 원칙
- 정확성이 먼저, 그다음이 쉬움이다. 둘이 부딪히면 정확성을 택하되 쉬운 말을 덧붙인다.
- 기본은 쉽고 짧게 설명한다. 어려운 용어는 풀어 쓴다.
- 정확한 이름(파일·함수·메뉴)이 행동에 꼭 필요할 때만 괄호로 함께 적는다.

### 구조
- 모든 답변은 `##` 또는 `###` 제목으로 구역을 나눈다.
- 항목이 3개 이상이면 불릿(-) 또는 번호(1.)를 쓴다.
- 비교·대조가 있으면 표를 쓴다.
- 절차·흐름·조건 분기가 짧은 글로 불명확할 때만 머메이드 흐름도를 쓴다.
- 계층 구조와 넘버링을 활용한다.

### 문장
- 한 불릿에는 한 가지 내용만 담는다. 두 가지 이상이면 하위 불릿으로 쪼갠다.
- 개조식 단문(짧은 끊어 쓰기)을 쓴다.
- 줄글 문단(3줄 이상 연속 서술)은 쓰지 않는다.
- 여러 동작을 한 문장에 "~했고 ~했으며 ~했다"로 이어 붙이지 않는다.

### 나쁜 예 (내부 함수명·구현 세부·여러 변경을 한 문장 줄글로 이어 붙임)
"CretopPage.collect_detail_map()이 OCR 원문 대신 섹션별 복사 텍스트 기준으로 품질검사를 하게 바꿨고 저장 로직도 함께 수정했습니다."

### 좋은 예
- 품질검사: 화면에서 복사한 글자 기준으로 바꿈
- 저장 부분: 함께 수정함
- 검색 부분: 그대로 둠

### 과정 중계 금지
- 당연한 절차(GBrain 읽기, 명령 실행, 파일 열기)는 멘트로 보고하지 않는다.
- "~하겠습니다" 예고, "~확인했습니다 / ~실행했습니다" 사후 보고를 쓰지 않는다.
- 명령 실행의 시작·완료는 보고하지 않는다. 아래 항목만 짧게 알린다.
  - 작업 분류
  - 선택한 대상(예: 파일·산출물)
  - 판단의 근거가 된 핵심 발견
  - 막힌 지점이나 사용자 결정이 필요한 부분
  - 문제 해결 원인 규명
